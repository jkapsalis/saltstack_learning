Configure the Salt Master (192.168.56.107)

1a — Verify Installation & Enable Services Master

You've already run dnf install salt-master. Confirm the binary is present, then enable and start the service:

# Verify the binary
salt-master --version

# Enable at boot and start immediately
systemctl enable --now salt-master

# Confirm it's running
systemctl status salt-master
# Expected: Active: active (running)
1b — Edit Master Configuration Master

Open /etc/salt/master and set the interface so the master listens on your host-only IP:

# Open the master config
vi /etc/salt/master

Find and uncomment / set these lines:

# Bind to your host-only adapter IP
interface: 192.168.56.107

# (Optional but recommended) give the master a resolvable name
# minions will use this to reach the master
publish_port: 4505
ret_port: 4506

# Where states/pillars will live
file_roots:
  base:
    - /srv/salt

pillar_roots:
  base:
    - /srv/pillar

After saving, create the directories and restart:

mkdir -p /srv/salt /srv/pillar
systemctl restart salt-master
1c — Open Firewall Ports Master

Minions contact the master on ports 4505 and 4506. Open them with firewalld:

firewall-cmd --permanent --add-port=4505/tcp
firewall-cmd --permanent --add-port=4506/tcp
firewall-cmd --reload

# Verify
firewall-cmd --list-ports
# Should show: 4505/tcp 4506/tcp
⚠
If you're using SELinux in enforcing mode, Salt ports are already labelled correctly in modern packages — but if connections fail, check ausearch -m avc -ts recent for denials.
1d — Add /etc/hosts Entry (recommended) Master + Minions

By default minions try to resolve the hostname salt. Adding an entry eliminates DNS dependency in your lab:

# On the MASTER:
echo "192.168.56.107  salt" >> /etc/hosts

# You will also add the same line on each minion VM after creation.
2

Set Up Salt Minions (new VMs)

💡
Repeat every sub-step below on each new VM you create. A quick way is to snapshot a "base" VM after step 2a, then clone it — you only need to change the hostname and minion ID.
2a — Install salt-minion on each VM Minion
# Bootstrap the Salt repo (same repo as master)
dnf install -y salt-minion

# Enable at boot (don't start yet — configure first)
systemctl enable salt-minion
2b — Configure /etc/salt/minion Minion

Edit /etc/salt/minion to point to your master and set a unique minion ID:

vi /etc/salt/minion
# Point to the master (by IP or hostname "salt" if /etc/hosts is set)
master: 192.168.56.107

# Give this minion a meaningful, unique ID
id: minion-01    # change to minion-02, web-01, db-01, etc.

Also add the /etc/hosts entry so "salt" resolves:

echo "192.168.56.107  salt" >> /etc/hosts

Now start the minion:

systemctl start salt-minion
systemctl status salt-minion
# Expected: Active: active (running)
2c — Accept Minion Keys on the Master Master

When a minion first connects, it sends a cryptographic key to the master for approval. Run this on the master:

# List all pending keys
salt-key -L
# Unaccepted Keys:
# minion-01

# Accept a single minion
salt-key -a minion-01

# OR accept ALL pending keys at once
salt-key -A

# Confirm accepted keys
salt-key -L
# Accepted Keys:
# minion-01
ℹ
In production, review each key before accepting. For your lab, salt-key -A is fine. You can also set auto_accept: True in /etc/salt/master to skip this step entirely during development.
2d — Test Connectivity Master

Run a quick ping to verify the master can reach all accepted minions:

# Ping all minions (returns True for each reachable minion)
salt '*' test.ping
minion-01:
    True

# Target a specific minion
salt 'minion-01' test.ping

# Get OS information from all minions
salt '*' grains.item os osrelease
✓
If test.ping returns True for every minion — Steps 1 and 2 are complete! You now have a fully operational Salt master with registered minions.

Quick Troubleshooting

Common Issues & Fixes

Minion key doesn't appear on master:

# Check minion logs for connection errors
journalctl -u salt-minion -n 50
# Usually: wrong master IP or firewall blocking 4505/4506

salt '*' hangs or returns nothing:

# Verify master is listening on both ports
ss -tlnp | grep -E '4505|4506'
# Test from minion side
telnet 192.168.56.107 4505
